# 実装設計書：Phase Plant Sampler ＋ 自動カット機能

## エグゼクティブサマリ  

**Phase Plant の Sampler** は単一サンプルを音源とし、MIDI ノートに合わせてピッチを変えつつ再生するジェネレータです。**Root（基音）調整、Offset（開始位置）、ループ(Start/Length/X-Fade)、Unison、モジュレーション、ポリフォニー管理**などを備え、**FLAC/WAV/AIFF** 形式のサンプルを読み込めます。サンプルエディタには正規化、DCオフセット除去、逆再生、ステレオ/モノ変換等の機能があります。一方、**Granular Generator** は短いグレイン再生用で、Sampler とは根本的に異なる粒状合成モジュールです。

本設計では Phase Plant Sampler 相当の機能に加え、**「無音/トランジェント検出による自動トリミング・スライス生成」**機能を統合したサンプラーを想定します。具体的には、読み込んだ波形から**無音部分やアタック位置を検出し、複数のスライス候補を自動生成**します。スライス境界には**ゼロ交差ポイント合わせや等パワー・クロスフェード**を適用し、クリックノイズを防止します。これらの機能はリアルタイム再生にも対応する必要があるため、**バッファサイズ制御、マルチスレッド化、SIMD 最適化**を用いてパフォーマンスを確保します。

**要点まとめ**：

- **対象機能**：Phase Plant Sampler の全機能（サンプル再生、ループ、Unison など）に加え、一般的な自動カット手法（無音検出、ピーク/トランジェント検出、BPM検出、機械学習法言及など）も網羅。  
- **機能比較表**：Sampler 機能、Granular の機能、サンプルエディタ/自動カット機能を表形式で一覧化。  
- **実装方針**：サンプル読み込み→音声バッファで保持→MIDI入力で再生→内部 Voice 管理（ポリフォニー、発音、盗用政策）→パラメータ（Root/Offset/Loop/XFade/Unison 等）による再生制御。  
- **アルゴリズム**：ピッチ変更のためのサンプルレート変換、アンチエイリアシング（帯域制限フィルタ）、リニア/キュービック補間、ループ処理（Infinite/Sustain/PingPong/Reverse）、等パワー・クロスフェード実装など。  
- **Unison/Voice**：Sampler 内部で複数の Sub-voice を同時生成し、Pitch/Detune/Spread で配置。グローバルポリフォニーでは最古・最弱音声を回収。モノモード時は Retrig/Legato をサポート。  
- **モジュレーション**：LFO/エンベロープ/マクロなどほぼ全パラメータに接続可能。Offset/Loop などをモジュレーションして新たなサウンド操作が可能。  
- **自動カット**：無音閾値法（フレームごとの RMS）、短時間エネルギー、スペクトル差分（スペクトルフラックス）、包絡の微分、ピーク比検出（例：高速/低速エンベロープの差分）などを実装し、閾値や最小長で調整。ビート検出はオンセットエンベロープ＋テンポ推定で行い、推定BPMに沿った均等スライスを追加。各手法に**利点・欠点・パラメータ・計算コスト**を明示し、精度要件別に推奨方式を示します。  
- **実装例**：C++（JUCEベース）、Rust（rodio/cpalなど）、Python（librosa, numpy）で主要機能（波形読み込み、無音検出、スライス生成、クロスフェード、ループ定義）を擬似コード＋簡易スニペットで提示。  
- **GUI/UX**：波形ビューワでスライスマーカー編集、ズーム/スナップ機能、バッチ処理、スライス単位のプレビュー、メタデータ（Root/Offset/Loop/Marker）保存を想定。フロー図・タイムライン図を Mermaid で示し、ユーザーの作業ステップを可視化。  
- **性能**：オーディオバッファレイテンシ（例：128サンプル ≈ 3ms）を目標に、オーディオスレッドと並列処理を分離。補間やフィルタは SIMD やマルチスレッドで最適化。大容量サンプルはメモリマッピングやストリーミング対応。  
- **テスト/評価**：自動カット精度は正解ラベルとの比較で Precision/Recall/F値を算出（例：MIREX オンセット検出手法参考）。性能ベンチは再生安定性、検出処理時間、メモリ使用量を測定。テストデータとしては既知のループ集やドラムループデータセットを用いる。  

以下、詳細に各機能・設計・アルゴリズム・実装例・GUIフローを解説する。必要に応じてコードスニペットや mermaid 図を含める。

## サンプラー機能詳細と実装方針

### 1. サンプル読み込み・波形表示
- **フォーマット**：Kilohearts 公式では工場付属はFLAC、ユーザは非圧縮 WAV/AIFF を読み込めると記載。実装では libsndfile や FFmpeg などで WAV/AIFF/FLAC をサポートします。
- **UI**：波形ビューを表示し、再生カーソル・選択範囲・スライスマーカーなどを描画できるようにします。JUCE の `AudioThumbnail` などを使って波形描画可能です。
- **Offset 設定**：再生開始サンプル位置をノーマライズされたパラメータ（0..1）で制御し、実際は `start_index = offset * sample_length` として実装します。GUIでは波形上でドラッグして設定可能にします。

```text
// Pseudocode: サンプル読み込み
sample = LoadAudioFile(path)  // PCMデータ、サンプリング周波数情報取得
```

```cpp
// C++ (JUCE) 波形読み込み例
AudioFormatManager fmtMgr;
fmtMgr.registerBasicFormats();
std::unique_ptr<AudioFormatReader> reader(fmtMgr.createReaderFor(File("sample.wav")));
AudioBuffer<float> buffer(reader->numChannels, (int)reader->lengthInSamples);
reader->read(&buffer, 0, (int)reader->lengthInSamples, 0, true, true);
```

```python
# Python (librosa) 波形読み込み例
import librosa
audio, sr = librosa.load('sample.wav', sr=None, mono=False)  # stereo の場合、返り値 (channels, samples)
```

### 2. Root（基音）設定
- サンプルの**基準ピッチ（ルートノート）**を設定し、MIDIノートとサンプル周波数の対応を決定します。`root_note`（MIDI番号）や`root_freq`を使い、MIDIノート再生時の再生レートを決めます。理想実装例：
  
  ```cpp
  // 擬似コード: 再生レート計算
  double target_freq = midiToFreq(played_note + tuning_cents/100);
  double root_freq = midiToFreq(root_note);
  double playback_rate = target_freq / root_freq;  // 2^((note - root)/12)
  phase_increment = playback_rate * (source_sr / output_sr);
  ```
  
- GUIではサンプル周期ごとに波形を拡大表示し、現在位置と1周期後の波形が一致するよう調整させます。これは目視チューニング支援です。

### 3. 再生・補間・アンチエイリアス
- **基本再生**：MIDI NoteOn でボイス生成。各ボイスは自サンプルの現在フェーズ（位置）を持ち、毎サンプル `phase += phase_increment` で読み進めます。
- **補間**：整数サンプル間では**線形補間**か**3次補間**、高品質なら**窓関数付きサンプル点対点 FIR**（windowed-sinc）を用います。例：1次線形補間は以下。

  ```cpp
  int idx = floor(phase);
  float frac = phase - idx;
  float out = sampleBuffer[idx] * (1-frac) + sampleBuffer[idx+1] * frac;
  ```

- **アンチエイリアス**：サンプル再生でピッチアップすると高周波成分が折り返すため、**バンドリミッティング**が必要です。高精度では、再生時に常にローパスフィルタをかける方法や、テーブルを複数用意する方法がありますが、複雑化するため、簡易版ではオーバーサンプリング（例：4倍）＋線形フィルタで代用しても良いでしょう。Kilohearts の内部詳細非公開ですが、類似実装では帯域制限やラウドネス保護が考慮されます。

### 4. Offset（オフセット）
- **Offset** はサンプル再生開始位置です。0.0～1.0 で正規化し、サンプル長に掛けて実インデックスを計算します。
  ```cpp
  double start_index = offset * sampleLength;  // offset in [0,1]
  phase = start_index;
  ```
- **モジュレーション対応**：Phase Plant の強みはほぼ全パラメータが LFO/Env 等で変動可能な点です。したがって Offset やループ位置もモジュレータのターゲットになります。設計では各パラメータに CV 値を加算する仕組みを用意し、オーディオレートモジュレーションも可能にします（ただし複雑な場合は低周波数で十分）。

### 5. ループ処理（Loop Mode）
- **Loop モード**：以下をサポートします。
  - **Infinite**: ループ区間を無限に前方向ループ。
  - **Sustain**: キーオン中はループし、キーオフでループ解除（そのまま再生継続）。
  - **PingPong**: ループ区間を前後に往復再生。
  - **Reverse**: ループ区間を後ろ向き再生。
- **ループ区間設定**：`loop_start` と `loop_length`（または終了点）を指定します。GUIでは Start/End 波形をズーム表示し、シームレスループ箇所を合わせる視覚補助を出します。
- **クロスフェード (X-Fade)**：ループ境界でのクリック防止のため、終端と先頭を重ねて**等パワー・クロスフェード**します。実装例（循環ループの場合）：

  ```cpp
  if (phase >= loopEnd) {
      // ループ終了直前からクロスフェード開始
      double n = phase - (loopEnd - xfade);
      double fade_t = n / (2*xfade);  // 0..1
      float a = cos(fade_t * M_PI/2), b = sin(fade_t * M_PI/2);
      float sampleA = interpolate(loopEnd - xfade + n);
      float sampleB = interpolate(loopStart + n);
      out = sampleA * a + sampleB * b;
      if (phase >= loopEnd) phase = loopStart + n;  // 先頭に巻き戻し
  }
  ```

  PingPong モード時は、前後それぞれの境界で同様に逆再生と正方向の混合処理が必要です。

### 6. Unison（ユニゾン）
- Sampler モジュール内にも独立した**オシレータ・ユニゾン**機能があります。設定できるパラメータは **Voice Count, Detune, Spread, Blend, Bias/Range** など。  
- **実装**：各ボイスが内部的に N 個の**サブボイス**を持ち、各サブボイスにデチューンとパンをかけてミックスします。基本的にはグローバルユニゾンより効率的に実装可能です。例構造：
  ```cpp
  struct SamplerSubVoice {
      double phase;
      double phaseIncrement;
      float pan;   // 左右定位
      float gain;  // ブレンドに応じたゲイン
  };

  struct SamplerVoice {
      int note;
      std::vector<SamplerSubVoice> subs;  // Unison 分
  };
  ```
- **モード**：`Hard/Smooth/Synthetic/Frequency Stack/Pitch Stack/Shepard/Chords` など複数モードに対応。それぞれサブボイスへのピッチシフト法や位相配置が異なります。例えば Hard/Smooth/Synthetic では全ボイス同音程でわずかにデチューンするのみ、Creative 系では倍音やオクターブ差を用いる特殊効果です。
- **Global Unison**：Phase Plant にはグローバルユニゾン設定もあり、これはジェネレータ全体を複製します。ここでは触れず、Sampler 内蔵ユニゾン実装に重点を置きます。

### 7. ポリフォニー／ボイス管理
- **ボイス割り当て**：Polyphony（最大同時音声数）を設定し、MIDI ノートオン時に新規ボイスを生成します。上限に達した場合は「最古かつ最弱のボイス」を盗用（再利用）します。つまり、発音中で最も昔に開始し、かつ音量が小さいボイスを停止させ、新たなノートに用います。  
- **モノモード**：Polyphony=1 で単音（モノフォニック）となり、Retrig/Legato モードを選択できます。Retrig では常にアタックがリスタート、Legato では保持中は再トリガーしません。  
- **構造例**：
  ```cpp
  struct VoiceManager {
      int maxPolyphony;
      std::vector<SamplerVoice> activeVoices;
      enum { StealOldestQuietest } stealPolicy;
      // ノートオン:
      void noteOn(int note, float velocity) {
          if (activeVoices.size() < maxPolyphony) {
              activeVoices.emplace_back(createVoice(note, velocity));
          } else {
              // 最古・最弱ボイスを再利用
              int idx = findOldestQuietest();
              activeVoices[idx] = createVoice(note, velocity);
          }
      }
  };
  ```
  
### 8. Sample Editor 機能
Phase Plant の Sample Editorにある機能は、主に**素材の整理・クリーニング用**です。実装例として以下を用意します：

- **選択ツール**：範囲選択し、ゲイン調整・フェードイン/アウトを実施。ユーザーが範囲をドラッグで指定し、ハンドルでフェードカーブを設定できるUIを提供します。
- **パンドラッグ・ズーム**：波形ビューのパン／ズーム操作。グリッドにスナップ機能。CTRL+ドラッグやホイールで実装可能。
- **ステレオモード**：L/R チャンネルの重ね表示・個別編集を切替。内部的にはチャンネル毎に同様の処理を行えばよい。
- **Normalize**：選択範囲 or 全体のピークを0dBにスケーリング。
- **Remove DC**：ゼロ平均化。全体平均値を引くか、高域通過フィルタを適用。
- **Reverse**：選択範囲 or 全体の波形を反転。
- **Mono/Stereo 変換**：一方方向にミックス／拡張。
- これらは GUI オペレーションで行われ、リアルタイム処理ではなくプリ処理と割り切ります。**自動カット**とも連携し、トリミング候補生成の一環として動作します。

### 9. Granular との違い
- Sampler（サンプラー）は単一サンプルを**ピッチ追従再生**するシンプルなサンプラーです。ループや Unison で音色を作り込みます。  
- 一方 **Granular Generator** はサンプルを短い**グレイン**（数ms～数十ms）に分割し、重ね合わせる粒状合成器です。サンプルの再生位置（Play Cursor）やグレイン長、出現率、Envelope、ランダム化パラメータなどがあり、全く別の操作性を持ちます。Granular ではサンプル波形全体を連続再生せず、あくまでグレイン単位で発音します。  
- 実装面でも異なり、Granular は各ノートで多数の粒（Voice内で複数グレイン発生）を扱うため、**リアルタイムで高密度な音素生成が必要**です。Sampler は基本的に1音＝1サンプルストリームです。

## 自動カット（スライス）アルゴリズム

**自動カット（スライス）**機能では、サンプル内の無音や目立つ音（トランジェント）を検出し、**スライスマーカー**を自動生成します。ここでは主な手法を列挙し、それぞれ長所・短所・パラメータ・計算コストなどを示します。

### 1. 無音検出（Silence Detection）
- **概要**：音声データの短いフレーム（例えば10ms）ごとに RMS（Root Mean Square）またはエネルギーを計算し、あらかじめ設定した閾値以下を「無音」とみなします。複数連続する無音領域が一定以上続けば、前後を分割してスライス候補とします。  
- **実装**：フレーム長とHopサイズ（オーバーラップ）を決め、窓付きFFTまたは直接時間領域で RMS を求める。閾値はデシベル（例：-40dB）で指定することが多い。閾値を高くすれば雑音までも無音と判定しやすくなるため、曲によって調整が必要です。  
- **利点**：単純で高速。静かなイントロ/アウトロや無音の区切りに有効。  
- **欠点**：バックグラウンドノイズやリバーブがあると検出精度低下。音が途切れていないループに向かない。細かなフレーズの切れ目を逃すことがある。  
- **パラメータ**：閾値（dB）、最小無音長（ms）、最小スライス長（ms）など。  
- **計算コスト**：非常に低い（O(N) で済み、大規模FFT不要）。  

RMS閾値法の例（openvpi/audio-slicer）では、「フレームごとのRMSを計算し、閾値以下を無音とする」と説明されている。フレーム内最小値付近をスライス境界として採用する手法も実装例で示されている。

### 2. 短時間エネルギー／スペクトル差分
- **短時間エネルギー**：無音検出と似ているが、**エネルギー勾配**に注目する方法です。フレームごとのエネルギー（または対数パワー）の増減を計算し、急な低下（音が消える瞬間）を検出します。  
- **スペクトル差分（スペクトルフラックス）**：周波数領域の変化量を用いる手法です。隣接フレーム間でスペクトルの変化量（L2ノルム差やマンハッタン距離）を計算し、大きな変化があればトランジェント/カット位置とみなします。**分離能**が高く、ピアノや弦楽器のアタックも検出しやすいが、FFTが必要なため計算コストは上がります。
- **利点**：無音ではなくても音の切れ目（エネルギーの急低下）を検出可能。楽器演奏など多彩な音源で有効。  
- **欠点**：背景ノイズや持続成分が多い場合はピークが判定しづらい。FFTサイズ・ホップサイズ調整が必要。  
- **パラメータ**：フレーム長・ホップ幅、スペクトル差分の閾値や平均化時間、低周波ノイズ除去フィルタなど。  
- **計算コスト**：FFT (例えば1024点) × O(N) が必要。リアルタイムでも近年のCPUなら可。  

### 3. トランジェント／ピーク検出
- **2つの包絡フォロワー法**：代表的な手法として、高速なエンベロープと遅いエンベロープの差分を用いる方法があります（SPL Transient Designerの「Differential Envelope Technology」）。高速アタックな包絡検出器（短いADSR）と緩やかな包絡を比較し、その差が閾値を超えるとトランジェントとみなす。  
- **ピアノロール法**：まず波形から総エネルギー包絡を抽出し、その上昇点を見つけるアプローチ。例えば12msごとの区間で最大値をプロットし、ピークと直前の谷を再帰的に細分化する手法も提案されている。  
- **FFTマルチバンド法**：複数バンド（バンドパスフィルタ or サブバンドFFT）でエネルギー変化を検出し、広帯域に現れる変化を確認する方法。  
- **利点**：楽器音の立ち上がり点をより精密に検出可能。  
- **欠点**：閾値設定や包絡フィルタ設計が難しい。リバーブ・レガート音の始まりでは検出困難。  
- **パラメータ**：短/長包絡の時定数、差分閾値、ピーク/谷の比率条件など。  
- **計算コスト**：包絡フォロワーは軽量だが、マルチバンドFFTは高い。

### 4. BPM/ビート同期スライス
- **概要**：ループ全体のテンポを推定し、推定BPMに合わせて**等間隔**にスライスする方法。例えばドラムループなど、一定拍子の素材に向いています。  
- **実装例**：Librosa の `beat_track` 関数では、オンセットエネルギーを計算→オートコリレーションでテンポ推定→動的計画法で最もテンポに合うビートを決定する。  
- **利点**：音楽的な拍に基づくスライスが可能。打ち込みやループ向け。  
- **欠点**：推定が間違うとスライス位置がずれる。非リズミカルな素材には不向き。処理時間もやや増大。  
- **パラメータ**：初期BPM推定、最小/最大BPM範囲、ホップサイズなど。  
- **計算コスト**：FFTベースのオンセット計算＋DP により、中程度のコスト。

### 5. 機械学習手法
- 近年は**CNNやRNN**を用いたオンセット検出や音事件検出モデルが研究されています。例えば教師あり学習でトランジェント位置を学習する手法など。ただし本設計では「参考情報」として簡単に触れるに留め、実装は従来手法中心とします。

### 6. 手法比較と推奨
- **高精度要求**：アタック位置まで厳密に切り分けたい場合は、トランジェント/ピーク検出（包絡差分、多段FFT）＋スペクトル差分が有効。ただしパラメータチューニングが難しく、ノイズに敏感。  
- **ざっくりスライス**：無音閾値法＋BPM同期で十分。音楽ループならBPM同期、ボーカルや環境音なら無音検出をまず試す。  
- **処理コスト**：リアルタイム制約が厳しい場合はフレームごとのエネルギー（RMS）処理が軽量。オフライン処理やバッチではFFT系も可。  
- **計算例**：1分の音声(44.1kHz)で30msフレームを使った場合、FFT(1024点) を使う手法は数十万回の演算が必要となるが、現代CPUならリアルタイム対応可能な程度です。

```cpp
// C++: 無音検出例 (単一チャンネル)
std::vector<int> detectSilence(const std::vector<float>& samples, int sr, float dbThresh= -40.0f, int minSilMs=300) {
    float ampThresh = pow(10, dbThresh/20.0f);
    int frameLen = sr * 10 / 1000; // 10ms
    int hop = frameLen;
    int silentCount = 0;
    std::vector<int> slicePoints;
    for (int i = 0; i + frameLen < samples.size(); i += hop) {
        float sum=0;
        for(int j=0; j<frameLen; j++) sum += samples[i+j]*samples[i+j];
        float rms = sqrt(sum/frameLen);
        bool silent = (rms < ampThresh);
        if(silent) silentCount += hop;
        else {
            if(silentCount >= minSilMs*sr/1000) {
                int slicePos = i - silentCount/2;
                slicePoints.push_back(slicePos);
            }
            silentCount = 0;
        }
    }
    return slicePoints;
}
```
```rust
// Rust: 無音検出（RustDSP crateなどがあれば簡単）
fn detect_silence(samples: &[f32], sr: usize, db_thresh: f32, min_sil_ms: f32) -> Vec<usize> {
    let amp_thresh = 10f32.powf(db_thresh/20.0);
    let frame = sr / 100; // 10ms frames at 44100Hz -> 441 samples
    let mut silent_len = 0;
    let mut slices = Vec::new();
    for chunk in samples.chunks(frame) {
        let rms = (chunk.iter().map(|x| x*x).sum::<f32>() / chunk.len() as f32).sqrt();
        if rms < amp_thresh {
            silent_len += chunk.len();
        } else {
            if silent_len >= min_sil_ms * sr as f32 / 1000.0 {
                let pos = (silent_len/2) as usize;
                slices.push(pos);
            }
            silent_len = 0;
        }
    }
    slices
}
```

// C++: 簡易トランジェント検出 (包絡フォロワー)
float alphaFast = 0.1f, alphaSlow = 0.01f;
float envFast=0, envSlow=0;
std::vector<int> transients;
for (int i = 0; i < samples.size(); ++i) {
    float x = fabs(samples[i]);
    envFast = (x > envFast) ? (alphaFast * (x - envFast) + envFast) : (alphaFast * (x - envFast) + envFast);
    envSlow = (x > envSlow) ? (alphaSlow * (x - envSlow) + envSlow) : (alphaSlow * (x - envSlow) + envSlow);
    if (envFast - envSlow > 0.02f) {  // 閾値例
        transients.push_back(i);
    }
}

## パフォーマンスとリアルタイム要件

- **バッファサイズ/レイテンシ**：オーディオ再生では通常 128～512サンプルのバッファを用い、約3～12msのレイテンシを目指します。サンプラーは主にオーディオスレッドで動作するため、補間・フィルタ処理はバッファごとに完了する必要があります。  
- **マルチスレッド**：スライス検出処理は負荷が高いので、非リアルタイムなバックグラウンドスレッドで実行すべきです。オーディオスレッドでは既存のマーカーに基づく再生のみを行い、検出中はUIにプログレスを表示します。  
- **SIMD最適化**：補間やフィルタ処理はデータ並列性が高いため、SSE/AVX 等で最適化可能です。JUCE のDSPモジュールや Eigen 等を使えば簡易にベクトル演算ができます。  
- **メモリ管理**：波形全体はRAMにバッファ化し、必要に応じてメモリマップ（大ファイル時）を検討。固定サイズバッファに不変なオーディオデータを読み込んでおけば、途中参照も高速。グローバルなキャッシュ（ルート選択時の周期波形スニペットなど）も利用可能です。  
- **マルチコア対応**：ボイス数が増えると各ボイスの処理が独立できるため、マルチスレッド化しても良いでしょう。ただしクロススレッドの同期は注意。  
- **ストリーミング**：非常に長いサンプル（数分以上）には、最初から全体読み込みせず逐次ストリーミング読み込みする設計もあり得ます（必要なら外部ライブラリ利用）。

## テストと評価

- **精度評価**：自動カットの精度は「検出されたスライス点 vs 正解マーカー」の**Precision/Recall/F値**で評価します。例えばあるループの本来のノートオン位置（人手ラベル）を正解とし、検出位置が±許容範囲内に入っていれば正解とカウントします。MIREXオンセット検出の手法評価を参考にするとよいでしょう。  
- **ベンチマーク**：単位時間あたりのサンプル処理数やスレッド数、CPU負荷を計測。50ms の音声に対して検出アルゴリズムが何ms で結果を出すか等。複数CPUコアでのスループット、メモリ使用量も測定します。  
- **テストデータ**：代表的なドラムループ、ギター/ピアノフレーズ、環境音素材など多様な音源で試験。公開データセットでは MIREX オンセット検出用のドラムやアンサンブル曲、GTZAN、SALAMI など。自作の場合はループライブラリ（例：Loopmasters等）も活用。精度別に推奨パラメータを調整します。
