# 汎用サンプラー必要調査計画

作成日: 2026-06-27

この文書は `docs/汎用サンプラー実装計画.md` を実装する前、または実装と並行して確認すべき調査項目をまとめたものです。実装判断に直結する一次情報、ライブラリ候補、ホスト互換性、性能評価を対象にします。

## 調査の目的

`docs/サンプラー.md` は Phase Plant Sampler 相当機能、自動カット、波形編集、GUI/UX まで広く含んでいます。実装前に全機能を一気に作るのではなく、次の不確実性を調査で潰してから段階実装します。

- どの機能を MVP に含めるべきか。
- DSP コアに入れるべきものと offline / UI 側に置くべきものは何か。
- Rust / WebAssembly で安全に使える decode / analysis ライブラリは何か。
- native VST3/CLAP と WebCLAP で sample asset をどう扱うべきか。
- auto slice の精度と速度をどの指標で合格にするか。
- plugin state に sample path、bank、metadata をどう保存するか。

## 調査成果物

各調査は、実装チケットに変換できる形で成果物を残します。

- 決定メモ: 採用 / 不採用 / 後回しと理由。
- 検証コマンド: 再実行できる command。
- 小さい fixture: 権利上問題ない wav / generated sample / JSON。
- 計測結果: load time、CPU、memory、bundle size。
- リスク: 実装前に回避策が必要なもの。

## R0: 現行実装の棚卸し

目的: 既存 VCSL 実装を汎用サンプラーへ拡張する前に、変更してよい層と守るべき挙動を整理します。

対象:

- `thirdparty/z-audio-dsp/crates/z-audio-dsp/src/sampler.rs`
- `thirdparty/z-audio-dsp/crates/z-audio-synth/src/vcsl_piano/`
- `crates/xtask/src/vcsl_piano.rs`
- `crates/z-audio-vcsl-piano-plugin/src/lib.rs`
- `crates/z-audio-webclap-vcsl-piano/src/lib.rs`
- `crates/wclap-plugin/src/lib.rs`

確認事項:

- `SamplerEngine` の既存 API を壊さず loop / reverse を足せるか。
- `SampleRegion` に field 追加した場合、既存 VCSL bank loader をどう versioning するか。
- `VcslSampleBank` を汎用型へ移行するか、汎用型を別追加するか。
- `EventKind` に MIDI CC や slice trigger を足す必要があるか。
- `ParamId` の空き範囲はどこか。
- WebCLAP wrapper の param / note event の制約は何か。

完了条件:

- 変更候補ファイル一覧が出ている。
- 既存 VCSL に影響する breaking change が明示されている。
- 汎用サンプラー初期 PR の最小差分が説明できる。

## R1: 参照仕様と機能範囲

目的: Phase Plant Sampler 相当と呼ぶために必要な機能を、公式または一次情報で確認します。

調査対象:

- Phase Plant Sampler / Sample Generator manual
- Phase Plant Unison / Voice 設定
- Phase Plant Sample Editor の保存仕様
- WAV `smpl` chunk、cue marker、loop metadata
- AIFF marker / instrument chunk
- SFZ v1 opcode subset

確認事項:

- Phase Plant の sampler が公式に対応する sample format。
- root note、offset、loop、xfade、save metadata の仕様。
- internal unison と global unison の違い。
- mono retrig / legato / voice stealing の期待動作。
- WAV / AIFF metadata をどこまで保存・読み込み対応すべきか。

採用判断:

- MVP に入れる機能と後続機能を分類する。
- 公式仕様が曖昧なものは、このプロジェクトの独自仕様として明記する。

完了条件:

- MVP / Phase 2 / 後続の機能一覧が確定している。
- 外部仕様に依存する実装は、参照 URL または仕様名を docs に残している。

## R2: audio decode / encode ライブラリ

目的: native と WebCLAP で使える audio file 読み込み方式を決めます。

候補:

- `symphonia`
- `hound`
- `claxon`
- `lewton`
- `rubato`
- platform decoder または external tool は原則後回し

確認事項:

- WAV / AIFF / FLAC 対応範囲。
- wasm32-unknown-unknown で使えるか。
- no_std / alloc / file I/O 制約。
- license。
- decode speed。
- channel layout と sample format 変換。
- metadata chunk を読めるか。
- encode が必要か、decode only でよいか。

推奨初期方針:

- MVP は offline / background decode に限定する。
- native plugin の audio thread で decode しない。
- WebCLAP はまず embedded bank を使い、browser file import は後続調査に回す。
- bank 内 PCM は f32 runtime / i16 on disk のどちらがよいかを計測で決める。

完了条件:

- WAV mono/stereo、FLAC、AIFF の小さい fixture で decode 成否が分かっている。
- wasm 対応可否が分かっている。
- 採用候補と fallback が決まっている。

## R3: bank / preset format

目的: 汎用サンプラーの bank と preset を長期的に壊れにくい形で設計します。

調査対象:

- 既存 `VCSLBANK` format
- SFZ subset
- SoundFont / SF2 の採用可否
- DecentSampler などの XML 系フォーマット
- JSON + binary payload の custom format
- WAV metadata (`smpl` / cue)

確認事項:

- bank に PCM を埋めるか、外部 sample path を参照するか。
- plugin state に absolute path を保存してよいか。
- project portable preset のために sample copy/import を実装するか。
- source hash、format version、metadata をどこへ置くか。
- WebCLAP asset と native asset で同じ format を使えるか。
- VCSL bank と汎用 bank を統合する必要があるか。

初期判断候補:

- `VCSLBANK` は既存互換のため維持する。
- 汎用版は別 magic の `ZSMPLBNK` として始める。
- preset は human-readable JSON、bank は binary とする。
- sample path は native state には保存するが、WebCLAP は embedded / bundle asset を基本にする。

完了条件:

- versioning 方針が決まっている。
- old bank を読めなくする変更を避ける設計になっている。
- state save/load の失敗時 UX が決まっている。

## R4: loop / interpolation / anti-aliasing

目的: click を避けながら loop / pitch playback の音質を確保します。

調査対象:

- linear interpolation
- 4-point Hermite / cubic interpolation
- windowed-sinc interpolation
- equal-power crossfade
- zero crossing snap
- pitch-up 時の anti-aliasing
- loop crossfade を realtime で行うか preprocess するか

確認事項:

- 現在の linear interpolation が MVP 音質として十分か。
- crossfade length を sample frames と ms のどちらで保存するか。
- loop start/end が sample 境界に近いときの clamp 方針。
- pingpong loop の phase 管理。
- reverse playback と offset / loop の相互作用。
- high pitch playback で aliasing が許容範囲か。

実験:

- sine / saw / piano sample の 12 半音上げ再生。
- loop 境界の max adjacent delta 計測。
- crossfade 0ms / 5ms / 20ms の比較。
- 44.1 / 48 / 96 kHz の output finite test。

完了条件:

- MVP interpolation が決まっている。
- high quality mode を後から足せる API になっている。
- loop click を検出する unit test 指標が決まっている。

## R5: auto trim / auto slice

目的: 無音検出、トランジェント検出、BPM 同期スライスの MVP と評価方法を決めます。

調査対象:

- RMS threshold による silence detection
- openvpi/audio-slicer の手法
- fast/slow envelope 差分
- spectral flux
- beat tracking / BPM estimation
- public onset detection evaluation methods
- simple generated fixtures

確認事項:

- MVP は RMS 無音検出だけで十分か。
- transient detection は envelope 差分までにするか、FFT を入れるか。
- BPM 推定を入れるか、固定 BPM grid slice だけにするか。
- marker の minimum distance、merge、dedupe 方針。
- zero crossing snap の探索窓。
- stereo sample の検出を L/R max、mid、sum のどれで行うか。
- analysis を native / xtask / WebCLAP のどこで実行するか。

評価指標:

- Precision / Recall / F1
- marker 許容誤差: 例 ±10ms / ±20ms
- 1分 audio の処理時間
- peak memory
- false positive marker 数

初期 fixture:

- 無音区切りつき synthetic pulse
- drum-like transient sequence
- sustain 成分が長い piano-like sample
- `docs/samples/piano.wav`

完了条件:

- MVP の auto slice algorithm が 1つ決まっている。
- threshold default と UI parameter が決まっている。
- expected marker fixture が追加できる状態になっている。

## R6: native plugin UI / state / background task

目的: DAW 上で sample import、解析、復元を安全に行う方法を決めます。

確認事項:

- `nih_plug` editor で file picker を使う方針。
- background task から plugin state へ bank を渡す同期方法。
- audio thread へ `Arc<SamplerBank>` を交換する方法。
- missing asset 状態を UI と音声処理にどう反映するか。
- host automation と sample editor parameter の境界。
- state save/load に sample path、bank hash、UI marker をどう保存するか。
- large bank load 中の playback をどうするか。

実験:

- small bank の background load。
- load 中に process が呼ばれ続けても glitch / panic がないか。
- state restore 時に missing path を検出できるか。

完了条件:

- native MVP UI の最小要件が決まっている。
- background load の同期 API が決まっている。
- state schema の草案がある。

## R7: WebCLAP asset / UI / host 互換性

目的: WebCLAP で汎用サンプラーを成立させるための asset 配布方式を決めます。

確認事項:

- `.wclap` bundle 内の `assets/` を wasm plugin から読めるか。
- 現在の `wclap-plugin` glue が file access を想定しているか。
- embedded bank の wasm size と load time。
- UI から plugin へ note event を送れるか。
- UI から file import できる場合、bank を wasm 側へ渡せるか。
- browser memory limit。
- state save/load の host 対応。

初期判断候補:

- MVP は embedded small bank。
- 汎用 WebCLAP file import は後続。
- bundle asset 読み込みが確認できたら `.wclap/assets/*.bank` へ移行する。

実験:

- dev bank size を 1MB / 5MB / 20MB に変えて load time を測る。
- `.wclap.tar.gz` size を記録する。
- WebCLAP host で MIDI input と parameter set/get を確認する。

完了条件:

- WebCLAP MVP の asset 方針が決まっている。
- bundle asset 方式の可否が分かっている。
- UI の最小通信仕様が決まっている。

## R8: performance / memory

目的: サンプラーとして現実的な polyphony、sample size、latency を決めます。

計測項目:

- 1 voice / 16 voices / 64 voices の CPU。
- linear / cubic interpolation の CPU 差。
- full preload bank の memory。
- bank load time。
- auto slice analysis time。
- WebCLAP wasm load time。
- `.wclap.tar.gz` size。
- native VST3/CLAP bundle size。

ベンチ条件:

- sample rate: 44.1 / 48 / 96 kHz
- block size: 64 / 128 / 512
- channel: mono / stereo
- sample length: 1s / 10s / 60s
- polyphony: 1 / 16 / 64 / 128

完了条件:

- 初期 default max polyphony が決まっている。
- full preload の上限目安がある。
- streaming を実装すべき閾値が決まっている。

## 優先順位

最初に必ず行う調査:

1. R0 現行実装の棚卸し
2. R2 audio decode / encode ライブラリ
3. R3 bank / preset format
4. R4 loop / interpolation / anti-aliasing の MVP 判断

auto slice は重要ですが、単音サンプラー MVP の成立後に進める方が安全です。R5 は Phase 4 実装前までに完了していればよいです。

## 実装を止める判断基準

次のどれかに当たる場合は、実装を止めて docs の決定メモを更新します。

- chosen bank format で native と WebCLAP の両方を支えられない。
- decode library が wasm で動かず、代替設計がない。
- plugin state に sample path を保存しても主要 host で復元できない。
- loop / xfade のクリック回避が unit test で安定しない。
- WebCLAP bundle size が MVP 用途でも現実的でない。
- audio thread に file I/O、decode、allocation が入りそうになる。
